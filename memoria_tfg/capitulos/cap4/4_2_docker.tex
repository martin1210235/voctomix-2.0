\section{Contenedorización con Docker Compose}
\label{sec:docker}

Docker~\cite{merkel2014docker,docker_compose_docs} permite empaquetar el sistema completo,
incluidas todas sus dependencias, en contenedores que se ejecutan de forma idéntica en
cualquier máquina con Docker instalado. El operador no necesita instalar GStreamer, Python
ni ninguna otra librería: con \texttt{make docker-up} el sistema arranca en segundos.
El despliegue se define en dos ficheros: \texttt{Dockerfile}, que describe cómo construir
la imagen base del sistema, y \texttt{docker-compose.yml}, que declara todos los servicios
y cómo se relacionan entre sí.

El \texttt{Dockerfile} original del repositorio databa de 2016 y usaba Ubuntu~15.10 con
imágenes que ya no existen. Se reescribió desde cero sobre Ubuntu~22.04~LTS. Las
dependencias se instalan en una sola capa para reducir el tamaño de la imagen: GStreamer~1.0
con todos sus plugins, FFmpeg~\cite{ffmpeg_project} para la inyección de fuentes de vídeo,
los bindings Python de GObject y la librería AMQP para la integración con RabbitMQ.
La imagen incluye un \textit{healthcheck} que comprueba si voctocore acepta conexiones
en el puerto~9999, de forma que Docker Compose espera a que el núcleo esté listo antes
de arrancar los servicios que dependen de él.

El principal desafío técnico de la contenedorización fue el aislamiento de red. En Docker,
cada contenedor tiene su propia interfaz de red, de modo que cuando un proceso se conecta
a \texttt{localhost} solo alcanza a los procesos de ese mismo contenedor. Esto supone un
problema para Voctomix porque las cámaras se conectan a voctocore mediante
\texttt{localhost:10000}. La solución adoptada es el \textbf{espacio de red compartido}:
todos los contenedores del sistema (cámaras, break, intro, stream blanker, audio manager
y telemetría) se configuran con \texttt{network\_mode: "service:voctocore"} en el fichero
Compose, lo que hace que compartan la misma interfaz de red que voctocore. Desde su perspectiva es como si todos
los procesos se ejecutasen en la misma máquina, sin necesidad de modificar ninguna línea
del código de Voctomix. RabbitMQ mantiene su propia red aislada y se comunica con el
servicio de telemetría mediante el protocolo AMQP.

Voctogui se ejecuta directamente en el equipo anfitrión y se conecta a voctocore a través
del puerto~9999. Ejecutar una interfaz gráfica dentro de un contenedor no es viable sin acceso al
servidor gráfico del sistema (X11 o Wayland), por lo que esta parte del sistema queda fuera de Docker. Los
recursos gráficos como logos e imágenes de fondo se comparten con el contenedor de
voctocore mediante un volumen, lo que permite modificarlos sin necesidad de reconstruir
la imagen.

Durante las pruebas se detectó un problema de arranque: las cámaras intentaban conectarse
a voctocore antes de que este estuviera listo, lo que producía fallos. La solución combina
dos medidas: si un contenedor falla, Docker lo reinicia automáticamente mediante
\texttt{restart: on-failure}; y antes de lanzar cada servicio, el sistema comprueba que el
puerto~9999 está disponible. Con estas medidas el despliegue completo resulta robusto
y repetible en cualquier máquina con Docker instalado.
