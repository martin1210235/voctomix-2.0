\section{Contenedorización con Docker Compose}
\label{sec:docker}

El despliegue con Docker~\cite{merkel2014docker,docker_compose_docs} se basa en
\textbf{una única imagen personalizada}, construida a partir del \texttt{Dockerfile}
sobre Ubuntu~22.04~\ac{LTS}. Esa imagen incluye todo lo necesario para ejecutar el
sistema: GStreamer~1.0 con sus \textit{plugins}, FFmpeg~\cite{ffmpeg_project}, las
librerías Python y \texttt{pika} para RabbitMQ. No hay una imagen por servicio ni
una imagen separada para el núcleo y para la interfaz: todos los contenedores del
\textit{stack} parten de esa misma imagen y ejecutan un proceso diferente sobre
ella, voctocore como núcleo de mezcla, los contenedores de cámara inyectando señal
con FFmpeg, el servicio de telemetría registrando eventos, etc.
Voctogui \textbf{no tiene imagen Docker}: corre directamente en el equipo anfitrión
porque necesita acceder al servidor gráfico del sistema para mostrar la interfaz.
La única imagen externa que se utiliza es la oficial de RabbitMQ
(\texttt{rabbitmq:3-management}), descargada automáticamente de Docker Hub.
La imagen propia se construye de forma automática la primera vez que se arranca
el sistema. Ambos ficheros de configuración se reproducen íntegramente en la
sección~C5 del Anexo~C.

Para arrancar el sistema completo basta con ejecutar \texttt{make docker-up}.
Internamente, el comando detiene cualquier contenedor previo para partir de un
estado limpio, levanta todos los servicios en segundo plano y espera a que
voctocore supere su verificación de salud (puerto~9999 disponible) antes de
continuar. Una vez el núcleo está listo, lanza voctogui de forma nativa en el
equipo y abre un monitor de programa en el puerto~15000. Al cerrar la sesión con
Ctrl+C, todos los procesos, tanto los contenedores como los procesos nativos, se
detienen de forma coordinada.

El principal problema técnico que hubo que resolver fue el aislamiento de red. En
Docker, cada contenedor está aislado y cuando se conecta a \texttt{localhost} solo
se comunica consigo mismo. Esto suponía un problema porque las cámaras necesitan
enviar la señal a voctocore a través de \texttt{localhost}. La solución fue hacer
que todos los contenedores compartan la red de voctocore mediante la opción
\texttt{network\_mode: "service:voctocore"} en el fichero Compose, de forma que
todos los procesos se ven entre sí como si estuvieran en la misma máquina, sin
modificar ninguna línea del código original.

Voctogui se conecta a voctocore a través del puerto~9999 y comparte los recursos
visuales como logos e imágenes de fondo mediante un volumen Docker, por lo que se
pueden modificar sin tener que reconstruir la imagen. Durante las pruebas se detectó
que algunos contenedores intentaban conectarse a voctocore antes de que estuviera
listo, lo que causaba fallos en el arranque. Para resolverlo, Docker reinicia
automáticamente cualquier contenedor que falle y antes de lanzar cada servicio se
verifica que el puerto~9999 está disponible, lo que garantiza un despliegue robusto
y repetible en cualquier máquina con Docker instalado
(figura~\ref{fig:escenario_docker}).
