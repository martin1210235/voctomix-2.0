\section{Contenedorización con Docker Compose}
\label{sec:docker}

Docker permite empaquetar el sistema completo, incluidas todas sus dependencias, en
contenedores que se ejecutan de forma idéntica en cualquier máquina con Docker instalado.
El operador no necesita instalar GStreamer, Python ni ninguna otra librería: con
\texttt{make docker-up} el sistema arranca en segundos.

El despliegue se define en dos ficheros: \texttt{Dockerfile}, que describe cómo construir
la imagen base del sistema, y \texttt{docker-compose.yml}, que declara todos los servicios
y cómo se relacionan entre sí.

\subsection{Reescritura del Dockerfile}

El \texttt{Dockerfile} original del repositorio databa de 2016 y usaba Ubuntu~15.10 con
imágenes de Docker Hub que ya no existen. Se reescribió desde cero sobre
Ubuntu~22.04~LTS, la versión de soporte extendido vigente.

Las dependencias se instalan en una sola capa para reducir el tamaño final de la imagen:
GStreamer~1.0 con todos sus plugins, FFmpeg para la inyección de fuentes de vídeo, los
bindings Python de GObject, la librería AMQP para la integración con RabbitMQ, y las
utilidades de red necesarias para las comprobaciones de puerto durante el arranque.

La imagen incluye un \textit{healthcheck} que comprueba si voctocore acepta conexiones
en el puerto 9999. Docker Compose espera a que esta sonda sea positiva antes de arrancar
los contenedores que dependen del núcleo, replicando en Docker la misma lógica de espera
activa que usan los scripts nativos.

\subsection{El problema de la red en Docker y su solución}
\label{sec:docker_red}

En Docker, cada contenedor tiene su propia red aislada: cuando un proceso dentro de
un contenedor se conecta a \texttt{localhost}, solo puede alcanzar a los demás procesos
del mismo contenedor. Esto es un problema para Voctomix porque las cámaras se conectan
a voctocore mediante \texttt{localhost:10000}. Si cada componente vive en su propio
contenedor, esas conexiones fallan.

La solución adoptada es el patrón de \textbf{espacio de red compartido}: los contenedores
de cámara y el servicio de telemetría se configuran con la opción
\texttt{network\_mode: "service:voctocore"} en el fichero Compose. Esto hace que esos
contenedores compartan la misma interfaz de red que voctocore, exactamente igual que si
todos los procesos se estuvieran ejecutando en la misma máquina. No fue necesario
modificar ninguna línea del código de Voctomix.

\subsection{Interfaz gráfica y recursos gráficos}

Voctogui se ejecuta de forma nativa en el sistema anfitrión y se conecta a voctocore a
través del puerto 9999 expuesto por Docker. Ejecutar una interfaz GTK dentro de un
contenedor introduce problemas de latencia y aceleración gráfica que la hacen inviable
en producción.

Los logos e imágenes de fondo que GStreamer necesita se montan dentro del contenedor
de voctocore mediante un volumen de tipo \textit{bind mount}: el directorio
\texttt{./images} del equipo anfitrión se expone como \texttt{/opt/images} dentro del
contenedor. Esto permite actualizar los recursos gráficos sin reconstruir la imagen.

\subsection{Resiliencia ante reinicios}

Durante las pruebas se observó que los contenedores de cámara intentaban conectarse a
voctocore antes de que este hubiera terminado de arrancar, produciendo fallos silenciosos.
El problema se resolvió combinando dos mecanismos:

\begin{itemize}
  \item \textbf{Reinicio automático:} la opción \texttt{restart: always} hace que Docker
        reinicie automáticamente cualquier contenedor que falle, sin intervención del
        operador.
  \item \textbf{Espera activa por puerto:} en los scripts de arranque nativos, los
        \texttt{sleep} fijos se sustituyeron por bucles que comprueban con
        \texttt{ss -lnt} que el puerto 9999 está en escucha antes de lanzar los
        servicios dependientes.
\end{itemize}
